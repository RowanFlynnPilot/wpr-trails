---
description: Add or update human-curated editorial values for a trail
argument-hint: <trail-id> (e.g. osm-yellow-trail, iat-dells-of-the-eau-claire)
---

Add or update an entry for trail `$ARGUMENTS` in `data/editorial.yaml`.

The human-curated `editorial.yaml` overrides the machine-generated
`editorial_auto.yaml` per-field, so I only need to specify fields where
I disagree with the auto-derived value.

## Steps

1. **Read both files for this trail**:
   - `data/processed/trails/$ARGUMENTS.json` for length, gain, max elevation, geometry context
   - `data/editorial_auto.yaml` to see what auto-derivation currently says, including
     the `_provenance` block explaining where each value came from
   - `data/editorial.yaml` to see if a human entry already exists

2. **Show me a summary** of:
   - Trail facts (name, length in miles, elevation gain in ft, county, managing authority)
   - Each auto-derived field with its provenance note
   - Any existing human entry

3. **Ask which fields I want to override**. Don't assume - the whole point of
   editorial.yaml is to capture nuance the algorithm can't see.

4. **Available fields** (only include ones where I'm overriding):
   - `difficulty`: easy | moderate | difficult | strenuous
   - `family_friendly`: true | false
   - `dog_policy`: allowed | leashed | prohibited | unknown
   - `bike_allowed`: true | false
   - `parking`: large | small | roadside | none | unknown
   - `accessibility`: wheelchair | stroller | none | unknown
   - `scenery_tags`: list from [overlook, water, glacial, prairie, old_growth, rocky_outcrop, wetland]
   - `seasonality`: list from [fall_color, wildflower, winter_xc, winter_snowshoe]
   - `mud_susceptibility`: low | moderate | high
   - `exposure`: sheltered | mixed | exposed
   - `drainage`: sandy | loamy | clay | mixed
   - `notes`: free text
   - `last_field_check`: ISO date (today's date if I just verified)

5. **Write the entry** into `data/editorial.yaml`, preserving existing entries
   and YAML formatting. Use YAML comments if a field warrants explanation.

6. **Run the scoring pipeline** to show the impact:
   ```
   python -m transforms.build_scores
   ```
   Then show the trail's new rank, score, and the top contributing factors.

## Don't

- Don't fill in fields I haven't given you a value for. Less is more here -
  editorial.yaml should only contain disagreements with auto-derivation, not
  duplicates of values we already agree with.
- Don't modify `editorial_auto.yaml` - that file is regenerated by the weekly
  pipeline and your edits would be lost.
- Don't change the trail's data/processed/trails/{id}.json - that's a build artifact.
